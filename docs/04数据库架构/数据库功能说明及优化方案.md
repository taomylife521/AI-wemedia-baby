# WeMedia X-Ray 数据库功能说明及优化方案

## 一、 数据库核心架构分析

### 1. 物理层封装 (`database.py`)

- **异步支持**：使用了 `aiosqlite` 提供完全异步的 SQLite 操作支持，提升了 I/O 密集型操作的响应能力。
- **性能调优**：启用了 WAL (Write-Ahead Logging) 模式以提升并发读取能力，并优化了 `cache_size`、`temp_store=MEMORY`、`synchronous=NORMAL` 等参数。另外加入了一个 Stage 1 优化机制：当 WAL 文件超过 10MB 时执行 Checkpoint 截断，控制磁盘空间与恢复速度。
- **并发兼容**：内置支持重试机制的查询方法 `execute_query`，以此应对 SQLite 高并发写入易产生的 `database is locked` 错误（指数退避策略）。
- **热备份**：提供了在线热备份机制 `backup_database` （使用 SQLite 的 `VACUUM INTO` 命令）。

### 2. 初始化与表结构 (`database_init.py`)

- **构建方式**：基于同步的 `sqlite3` API 进行了一次性的建库建表操作，所有建表脚本直接编码在 Python 主脚本中。并为主外键、常查字段创建了较完善的索引。
- **核心数据模型**：当前系统中定义了十多张表，包括：
  - `users`：用户体系
  - `subscriptions`：订阅管理
  - `platform_accounts`：各内容平台多账号、身份与 Cookie 状态管理
  - `publish_records`：各类平台的发布记录状态追踪
  - `batch_tasks` / `batch_task_executions`：批量操作任务相关记录
  - `media_files`：系统内的多媒体文件素材资源管理

### 3. 数据层与业务层融合模型

- 虽然在抽象上（`domain/models/` 等目录）梳理了独立的业务实体模型（如 `account.py`, `publish_task.py` 等），但是在持久化存储实现上，所有实体的增、删、改、查（CURD）逻辑目前完全集中在一个 `AsyncDataStorage` 类中。

## 二、 潜在 Bug 及隐患分析 (代码审计发现)

### 1. 事务无重试机制 (严重性：高)

- **问题表现**：在 `execute_query` 方法中针对 SQLite `database is locked` 实现了指数退避重试，但在负责跨表或多步更新的 **`execute_transaction` （执行多条语句构成的事务）中完全没有重试逻辑**。
- **触发风险**：当系统出现并发回调更新（如多个爬虫或者渲染任务并发写入状态）且触发了文件锁时，单条查询 `execute_query` 可能重试成功，但事务 `execute_transaction` 会直接抛出异常，导致数据状态更新失败甚至产生脏数据（尽管有 rollback，但业务流程可能因此中断）。

### 2. 上帝类反模式 (严重性：中/长远影响大)

- **问题表现**：`AsyncDataStorage` 类已膨胀至一千多行，它不仅负责底层的连接管理，还要充当所有业务表操作的代理人。无论是“创建平台账号”、“处理发布记录”还是“记录登录日志”所有的 SQL 全部硬编码在其中。
- **触发风险**：这严重违背了“单一职责原则 (SRP)”。当开发新插件和新功能需要扩充字段时，所有人都需要修改这个庞大的类。这会导致严重的合并冲突，并且代码可读性和可测试性较差。

### 3. 动态拼接条件查询存在脆弱性 (严重性：低)

- **问题表现**：在类似 `get_publish_records` 等动态过滤器方法中，采用了基于 List 的 `conditions.append` 与 `params.append`，最后采用 `AND` 拼接。
- **触发风险**：虽然利用了参数化绑定防范了 SQL 注入风险，但人工管理参数顺序和变量对应关系极易出错。特别是字段逐渐增多时，稍不留神就会导致位置不匹配。

### 4. 原始的表结构升级策略 (严重性：中)

- **问题表现**：在 `database_init.py` 以及 `database.py` 中，一旦要在旧表上加字段，使用的是 `try...except sqlite3.OperationalError: ALTER TABLE...` 这种异常拦截补偿策略。
- **触发风险**：当应用升级使得表结构变化越来越大（如字段改名、类型变换）时，SQLite 这类不能原生地支持复杂 `ALTER` （需要创建临时表、导数据再更名）的数据库非常容易翻车。这种简易脚手架脚本将变得极其难以维护，容易导致用户从旧版本升级时数据损坏。

## 三、 优化方案建议

### 1. 架构解耦：引入 Repository (仓储) 模式

- **实施说明**：将庞大的上帝类 `AsyncDataStorage` 大幅瘦身，回归其“基础设施”的角色——仅负责连接池管理、环境调优、执行通用查询和事务（DB Context 的角色）。
- **执行计划**：在 `src/infrastructure/storage/` 目录下新增各类 Repository，如 `UserRepository`、`PublishRecordRepository`、`PlatformAccountRepository`。各自分管对应的业务领域的 SQL 拼装和对象映射，并在需要跨表操作时共享底层的事务下发。这会让各板块指责变得分明，极大改善可维护性。

### 2. 补齐并发与事务级重试补丁

- **实施说明**：将当前 `execute_query` 中的锁冲突（`Locked`）指数退避重试逻辑提取为一个通用的异步装饰器，例如 `@retry_on_sql_locked`。
- **执行计划**：不仅在单条查询上，更应将此装饰器挂载在 `execute_transaction` 方法外侧。从而保证 SQLite 偶发性的文件系统锁定不会造成整个业务链路熔断。

### 3. 稳妥管理数据模型：引入 Alembic 工具

- **实施说明**：告别“Try-Catch 加字段”的原始方式，引入成熟的数据库迁移工具。
- **执行计划**：配置 **Alembic** 等迁移方案。记录每次表结构的变化为迁移版本脚本（Migrate Versions）。这能保障安装不同老版本客户端的用户在更新程序时，能够安全、顺滑且可降级地更新表模式。

### 4. SQL 编写优化：构建轻量级 QueryBuilder

- **实施说明**：如果在保持原生 SQL （不使用 SQLAlchemy 等重型 ORM）的前提下，为了防止多条件查询手工拼接失误，建议实现一个轻量型的 Query Builder 工具类。
- **执行计划**：使用 Query Builder 生成统一的 `SELECT * FROM table WHERE ...` 结构，自动确保查询参数和绑定符的严格对应关系，杜绝开发中的粗心失误。
