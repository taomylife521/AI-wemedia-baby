# 更好的数据库系统重构方案评估

基于此前对当前系统（`aiosqlite` + 纯手工 SQL + 集中式上帝类）的分析，针对 WeMedia X-Ray 项目作为一个**异步桌面客户端/本地自动化工具**的业务特性，数据库在底层引擎上**继续保持 SQLite 是绝对正确的设计**（零配置、无需搭建服务、方便打包分发）。

但**数据访问层 (Data Access Layer)** 需要全面摒弃手工拼接 SQL 和 `Try-Catch` 增删字段的做法。

以下是三种在 Python 异步生态中最主流且能彻底解决当前痛点的“更好方案”评估：

---

## 方案一：Tortoise ORM (🏆 综合最推荐)

Tortoise ORM 是一个易用的、原生的异步 ORM，其设计灵感来源于非常流行的 Django ORM。

### 优势

1. **原生且彻底的异步**：无需像其他框架那样通过一层包装，从头到尾全是异步，与当前项目全面拥抱 `asyncio` 的理念完美契合。
2. **极致的开发体验**：代码极其简洁，查询数据只需要 `await User.filter(username="test").first()`，彻底摆脱手工拼装 SQL 的噩梦。
3. **内置自动迁移工具 (Aerich)**：修改了类的字段（比如加了个 `avatar_url`），只需要运行一行命令，就会自动生成并执行安全的 ALTER TABLE 升级脚本。用户端旧版升级新版时平滑且无痛。
4. **轻量级**：学习曲线平缓，适合当前项目十几个表的规模。

### 劣势

- 如果业务场景涉及极度复杂的联表查询、窗口函数，它的表达能力不如 SQLAlchemy 强大（但本项目主要是做简单的 CURD 和状态流转，完全够用）。

---

## 方案二：SQLModel + Alembic (🚀 最现代/类型提示友好)

SQLModel 是由 FastAPI 作者主导开发的现代 ORM，它将 Pydantic（数据校验神器）和 SQLAlchemy（老牌 ORM 引擎）完美融合在一起。

### 优势

1. **完美的类型安全约束**：配合 IDE，你在输入查询结果或写入对象时拥有极佳的代码提示，不会再出现“字典缺少某个键”引发的运行时崩溃。
2. **底层极其强大**：它本质上是 SQLAlchemy 2.0，所以具备最强的 SQL 表达能力。
3. **与现代栈契合**：如果本项目未来计划将部分功能抽离出 API 服务提供给各种插件调用，基于 Pydantic 的 SQLModel 能让你省去一多半的序列化（JSON转换）代码。

### 劣势

- **异步配置较繁琐**：因为基于 SQLAlchemy 包装，配置其全异步（AsyncSession）的样板代码较多。
- Alembic 迁移环境配置相比 Tortoise 的 Aerich 要略显硬核。

---

## 方案三：SQLAlchemy 2.0 (Async) + Alembic (🏗️ 工业级重型标准)

这是 Python 界绝对的重型工业标准，无论多大厂、多复杂的 SaaS 都在用。

### 优势

- **万能**：支持数据库集群、读写分离、最高级的锁控制和任意复杂的 SQL 方言。
- 生态最大，遇到任何疑难杂症几乎都能秒搜到答案。

### 劣势

- **太重了（杀鸡用牛刀）**：对于一个桌面自动化分发工具而言，引入完整的 SQLAlchemy 等同于让一个卡车引擎装在了家用轿车上。
- **陡峭的学习曲线**：2.0 异步版本的写法变动较大，新旧风格混杂容易给团队后续开发带来负担。

---

## 🎯 总结与推演决策

既然目前的痛点是**“代码太长太乱”**和**“改表结构怕丢数据”**：

- 如果团队希望**最快速度重构、代码最简洁、马上见效**：请选择 **方案一 (Tortoise ORM)**。
- 如果团队在项目中已经大量使用了 **Pydantic** 用于类型检查和校验，希望保持技术栈高度统一：请选择 **方案二 (SQLModel)**。

无论选择上述哪一种，**Repository (仓储层设计)** 的重构依然是必须的（即将数据库操作放入 `UserRepository`、`PublishRecordRepository` 中，使其与具体的 ORM 分离），以保障未来哪怕再换引擎也不需要改上层业务逻辑。
