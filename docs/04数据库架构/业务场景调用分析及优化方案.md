# WeMedia X-Ray: 数据库业务场景调用分析及优化方案

通过对 `wemedia-baby` 核心代码的全局检索与阅读，我们深入分析了当前项目（特别是**自媒体账号管理**和**发布任务记录**两大核心场景）对数据库调用的真实现状。

## 一、 当前数据库使用场景现状

### 1. 账号记录场景 (Account Management)

**功能描述**：用于保存已登录自媒体账号的状态、平台类型、Cookie 路径信息以及平台昵称等。

**调用分布**：

- **UI/页面直连**：`src/ui/pages/account/view.py`, `src/ui/pages/workspace_page.py` 等诸多 GUI 页面直接通过 `service_locator.get(AsyncDataStorage)` 获取底层数据库实例，然后执行 `get_platform_accounts()`。
- **ViewModel**：`account_viewmodel_async.py` 内部同样直接持有底层 `AsyncDataStorage` 对象进行状态同步。
- **Service 层**：`account_service.py` 及 `account_group_service.py` 内部也直接调用了底层的增删改查方法。

### 2. 发布任务场景 (Publish Records & Batch Tasks)

**功能描述**：记录用户提交的分发任务（如视频、图文）、分发状态（pending, running, success, failed）、目标平台以及定时任务等。

**调用分布**：

- **发布核心页面**：`publish_records_page.py`, `single_publish_page.py`, `publish_list_page.py` 等 UI 控件中，直接大量混杂了对 `data_storage.get_publish_records` 或 `data_storage.update_publish_record_content` 的调用。UI 控件甚至负责处理数据库如果查不到数据的 fallback 逻辑。
- **后台管道/任务调度**：`src/infrastructure/common/pipeline/publish_pipeline.py`（后台发布管道）中，发布引擎直接和 `AsyncDataStorage` 对接，获取 pending 状态记录并改变其为 running。

---

## 二、 核心痛点与问题诊断

通过对上述场景的分析，应用中暴露出一个严重的架构问题：**Smart UI (智能视图) 反模式与严重的层级穿透**。

1. **过度的 UI 耦合 (UI 直连 DB)**
   当 `LoginDialog` 或者是 `PublishListPage` 想展示列表或改变状态时，它们绕过了业务逻辑层，直接拿到了底层的 DB 实例执行 SQL 查询方法的封装。
   - **风险**：只要对数据库表加一个字段，或者对查询逻辑做一个修改，无数个 UI 文件都需要跟着改。UI 代码极难编写单元测试。
2. **缺乏缓存与统一拦截可能**
   由于整个项目的几十个页面和后台任务直接并发请求同一个 `AsyncDataStorage`，一旦业务规模变大，想在“获取账号列表”上加个防抖、加个内存缓存、或者加个权限校验，根本无从下手——因为调用方漫山遍野。
3. **数据模型割裂**
   在 `src/domain/models/` 定义的 `Account` 类型经常得不到充分利用。UI 端常常直接拿回一层含有数据库原始字典数据 (`dict`) 的列表，然后再手工转化，极易产生 KeyError。

---

## 三、 结合业务场景的架构优化方案

为了解决 UI 深度绑定数据库的死结，结合上一篇报告中建议引入更好的 ORM 工具（如 Tortoise ORM 等），建议在**账号管理**和**发布管道**等场景实施以下重构计划：

### 阶段一：抽离 Repository (仓储层) 和 DTO (数据传输对象)

不再让任何业务代码认识 `AsyncDataStorage`。
针对“账号管理场景”，提供统一的 `AccountRepository` 接口：

```python
# 将数据访问黑盒化
class AccountRepository:
    async def get_active_accounts_by_platform(self, user_id: int, platform: str) -> List[AccountDTO]:
        # 内部使用 ORM 查表并返回经过严格类型校验的数据模型，不再是原始的 SQLite 字典！
        pass
```

### 阶段二：严格的三层架构职责划分 (MVP/MVVM 的真正落地)

强制隔离 UI 和 Database。所有的视图逻辑只能调用 Service/UseCase，Service 调用 Repository：

#### 以“展示发布记录”场景为例，重构后：

1. **DB 层 (ORM)**: 负责把 SQLite 中的 `publish_records` 转化为。
2. **Service 层 (`PublishService`)**: 提供 `get_user_publish_history(user_id)`，如果需要缓存、日志统计，都在这里统一加上。
3. **UI/ViewModel 层 (`PublishRecordsPage`)**: 只注入调用 `PublishService`。拿到格式化好的 DTO 后，专门负责将数据显示到 PySide/PyQt 控件树上。

### 阶段三：分离后台调度器与前端展示

对于 `PublishPipeline`（后台视频上传引擎），它不仅要查数据库改状态，还需要在任务完成后**通知前端 UI 刷新**。

- **优化点**：引入**事件总线 (Event Bus)** 或 PyQt 的 `Signal/Slot` 机制。当发布状态被 Backend 改变（如成功），触发一个 `TaskCompletedEvent`，前端的 PublishRecordsPage 监听到事件后，向 Service 请求取最新一条数据并局部刷新 UI，从而彻底抛弃前端页面轮询数据库的设计。

---

## 四、 实施路径建议

鉴于项目庞大，建议分步实施：

1. **试点重构**：首先挑选**订阅管理 (Subscription)** 或业务最简单的 **用户登录注册** 场景，使用 **「ORM -> Repository -> Service -> ViewModel」** 全新链路打样。
2. **核心业务替换**：再集中精力将最复杂的 **发布记录管理** 和 **多平台账号管理** 从 UI 中剥离，收拢至专门的 `PublishManager` 和 `AccountManager` 中。
3. **老代码下线**：彻底移除 UI 层中所有 `service_locator.get(AsyncDataStorage)` 的代码，斩断层级穿透。
