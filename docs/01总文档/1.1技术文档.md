## 1. 项目概述

### 1.1 软件简介

**媒小宝 (WeMediaBaby)** 是一款基于 **整洁架构与平台插件化架构** 设计的自媒体多矩阵账号自动化管理与多发工具。采用“核心开源 + 闭源扩展”的混合部署模式，旨在为创作者和工作室提供隔离、安全、高效的内容分发流。

### 1.2 核心价值

- **插件化架构**：通过 `src/plugins` 接口层实现多平台隔离，支持开源社区贡献与商业平台脚本的热插拔，已深度支持视频及图文的自动化封装和封面提取。
- **双核浏览器引擎**：
  - **内嵌模式** (`QWebEngine`): 提供轻量级的界面载入，如登录二维码扫码。
  - **自动化模式** (`PlaywrightBrowserService`): 提供数据完全物理隔离环境下的自动化操作（上传、发布），具备深层次指纹混淆 (`Undetected`) 及风控绕过能力。
- **业务解耦与 MVVM**：彻底剥离底层逻辑与表现层，通过视图模型（ViewModel）层进行双向数据状态绑定管理，保障大型应用的稳定与扩展性。
- **X-Ray 数据日志探针**：基于 WeMedia X-Ray 工具集的重构，为底层分析各平台（如抖音、快手）协议树、发布流程反馈和页面结构提供强有力的日志观测支持，赋能自动化与 AI 开发。

## 2. 系统架构

### 2.1 分层架构设计 (5层模型)

项目基于领域驱动设计 (DDD) 的思想重构，划分为以下层次：

```mermaid
graph TD
    subgraph "界面展现层 (UI Layer)"
        UI[PySide6 + FluentWidgets]
        Pages[独立业务页 Pages]
        ViewModels[ViewModel (视图状态映射)]
        Dialogs[通用弹窗框架]
        UI --> ViewModels
    end

    subgraph "服务编排层 (Service Layer)"
        ViewModels --> Services
        Services[Services (Account, Publish, Workspace等)]
        BrowserService[Browser/Playwright调度器]
        Services --> Domain
        Services --> Plugins
    end

    subgraph "领域模型层 (Domain Layer)"
        Domain[Models 业务抽象实体]
        Repos[Repositories Interface (数据处理网关约束)]
    end

    subgraph "核心插件层 (Plugin Layer)"
        PluginMgr[平台总加载与分发器]
        Interfaces[Interfaces 标准(Login/Publish规约)]
        Impl_Open[社区插件 (抖音/快手等)]
        PluginMgr --> Interfaces
        PluginMgr --> Impl_Open
    end

    subgraph "基础设施底座 (Infrastructure Layer)"
        Infra[存储系统, 网络缓存管理, 日志收集]
        Tortoise[Tortoise ORM (应用持久化)]
        BrowserFactory[浏览器特征及实例工厂]
        BrowserFactory --> Playwright
        Services --> Infra
        Plugins --> Infra
    end
```

### 2.2 核心目录结构与作用

请参考配套文档 **《1.7代码文件结构说明.md》** 以获取到具体代码文件级别的详细目录树作用清单。总体划分如下：

- **`domain/`**：系统的最核心模型（如账号、发布任务特征等），没有任何外部框架依赖。
- **`services/`**：应用的核心业务逻辑中心，统筹编排从登录认证、持久化存储到执行发布的任务下发。
- **`infrastructure/`**：包含数据库 ORM 底层通信、原生 Playwright 服务调度及进程维持体系。
- **`ui/`**：包含主窗口、分栏路由配置、ViewModel 及 FluentUI 组件库映射。
- **`plugins/`**：定义好核心抽象后，实现与各大自媒体视频平台进行原生 DOM 交互和协议登录的具体脚本。

## 3. 核心技术栈

### 3.1 依赖总览

| 分类             | 核心技术选型             | 核心用途                                               |
| :--------------- | :----------------------- | :----------------------------------------------------- |
| **基础 GUI**     | PySide6                  | Qt for Python，负责本地窗口渲染                        |
| **设计语言**     | PySide6-Fluent-Widgets   | 现代化 Win11 风格的组件框架支持                        |
| **异步集成**     | qasync                   | Qt主事件循环与 Python asyncio事件循环无缝融合方案      |
| **应用数据库**   | Tortoise ORM / aiosqlite | 强基于协程机制的类型推断 ORM 框架，管理本地持久化存储  |
| **通信与 IO**    | aiohttp / aiofiles       | 非阻塞网络交互与本地媒体读写控制                       |
| **浏览器操控**   | playwright               | 实现底层协议交互的核心自动化基石                       |
| **风控与反检测** | undetected-playwright    | 重写底层通讯并覆写注入 Canvas/WebGL 等隐蔽特征对抗识别 |
| **环境加密**     | cryptography / keyring   | 对于系统级证书凭据与用户关键脱机 Cookie 序列的保密保护 |

### 3.2 架构开发规范

1.  **全链路异步化**：从文件读写、ORM层查询到网络封包拦截，底层均基于 `async/await`，坚决拒绝对主线程事件循环产生任何阻塞。
2.  **解耦数据流**：禁止任何 ViewModel或服务层之间跨级互相直接持有强引用通信。由顶向下调用，事件通知依赖总线。
3.  **防崩溃兜底**：UI 层对于所有预期外的底层指令必须持有 `Exception` 的大粒度兜底捕获并提供给侧边栏状态灯展示。

## 4. 核心功能及模块机制

### 4.1 数据隔离与工作区引擎 (`src/services/workspace`)

设计了严格的工作区软物理隔离体制。工作室环境运营往往涉及多批不同定位的主体号，通过切换 Workspace，可以实现：

- 独立的底库数据记录。
- 独立的代理网路、浏览器 UA 集以及本地独立缓存目录（Profile）。

### 4.2 Playwright 服务治理中心 (`src/infrastructure/browser`)

区别于通常的爬虫脚本一头到底执行，将 Playwright 后置剥离成为后台守护服务：

1.  **特征重构工厂**：通过工厂类可以按照不同平台账号的过往配置分阶段无休止拉起带有特定 UA 标识和预置伪装插件池的窗口。
2.  **无状态唤起与重用复用**：系统记录每组 Context 连接点，对频繁操作同一个平台的多个任务管道允许实现引擎会话（Session）保持与连接复用，极大提升执行流畅度并避免验证码风控。

### 4.3 插件双生模式机制 (`src/plugins`)

采用 `Protocol` 标准的结构性鸭子类型管理：

- 开发新媒体平台只需继承特定的基石类型如 `AbstractLoginPlugin` 与 `AbstractPublishPlugin`。
- 将选择器与 DOM 操作分离至 `selectors.py`，因为平台前台往往变动高频，此机制通过将结构路径文本化以允许系统可以实现在不用打包重新分发核心应用库的前提下进行云端“热更新规则注入”。

### 4.4 状态同步呈现引擎 (`src/ui/viewmodels`)

不再由窗体回调函数内自己管理自己的拉取和载入：

- 创建类似 `account_viewmodel_async.py` 这类管理类承担“大脑状态机”。
- View 只需要执行简单的基于信号的纯粹视图渲染：比如当 ViewModel 收到了“有一个账号 Cookie 过期”底层消息变更时，View端接收到响应的信号仅需通过 ID 将表格的一行渲染成红牌报警颜色。

## 5. 常规部署构建

1. **环境准备与依赖安装**：通过 Python 标准的基于 `requirements.txt` 下沉，并需要额外执行 `playwright install chromium` 操作装配底层二进制依赖。
2. **源码打包机制**：依靠 PyInstaller 构建初版，或采用 Nuitka 转译为 C代码级库构建最终发行版进行性能优化和源码物理加密隐蔽。
