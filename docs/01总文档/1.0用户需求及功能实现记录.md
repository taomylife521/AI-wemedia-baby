## 1. 项目概述 (Project Overview)

**媒小宝** 是一款基于 Windows 的桌面端自媒体管理软件，旨在通过自动化技术帮助创作者高效管理多平台账号与内容发布。

### 1.1 核心价值

- **多平台聚合**: 统一管理抖音、快手、小红书、视频号等主流平台。
- **安全隔离**: 采用嵌入式浏览器技术，Cookie 本地加密存储，实现多账号环境隔离。
- **自动化操作**: 模拟人工操作发布内容，降低重复劳动，同时降低封号风险。
- **混合架构**: 核心框架开源，高级功能（企业级插件）可扩展。

### 1.2 技术架构 (Technical Architecture)

- **GUI 框架**: Python + PySide6 (Qt for Python)，采用 `QFluentWidgets` 实现现代化 UI。
- **核心逻辑**: DDD (领域驱动设计) 分层架构，分离 UI、Service、Domain、Infrastructure。
- **浏览器引擎**: 采用 `Playwright` + `Undetected-Playwright`，实现抗检测的网页自动化。
- **插件系统**: 独创的插件化架构 (`src/plugins`), 支持动态加载开源及闭源插件。
- **数据存储**: `aiosqlite` (异步 SQLite) + `Fernet` (加密) + `JSON` (配置)。

## 3. 详细需求说明 (Detailed Requirements)

### 3.1 账号与权限 (Account & Auth)

**需求**:

- 软件本身需要账号体系（User），用于同步配置和订阅状态（云端）。
- 平台账号（PlatformAccount）依赖 Cookie 本地存储，**严禁上传云端**。
- **现状**:
  - [x] 本地账号数据库设计。
  - [x] Cookie 本地加密存储。
  - [ ] 软件登录/注册/权限鉴权（目前为单机直连）。

### 3.2 浏览器自动化体系 (Browser Automation)

**核心重构点**:

- 摒弃了早期直接在 UI 调用 Playwright 的方式。
- **现在**: UI 通过 Signals 发送指令给 `PlaywrightBrowserService`。
- **流程**:
  1. 用户点击“更新 Cookie”。
  2. Service 启动无头/有头浏览器。
  3. Service 注入指纹与环境。
  4. 用户手动操作/脚本自动执行。
  5. Service 捕获 Cookie 变化并持久化。

### 3.3 发布系统 (Publishing System)

**需求**:

- 支持视频、图文发布。
- 支持定时任务。
- 支持多账号并发（需考虑风控）。
- **流程**:
  1. 用户创建 `PublishTask`。
  2. `TaskScheduler` 调度任务。
  3. 调用对应平台的 `PublishPlugin`。
  4. 插件驱动浏览器完成上传、填写元数据、点击发布。
  5. 反馈结果至数据库。

### 3.4 订阅与商业化 (Subscription - Future)

**规划**:

- 免费版：限制账号数量/并发数。
- 订阅版：解锁 Pro 插件（小红书/视频号）、无限制账号。
- **实现路径**:
  - 接入微信/支付宝支付 SDK。
  - 服务端验证许可证。


刷新登录状态功能介绍：点击刷新登录状态后，判断已登录账号登录状态为在线还是离线。

刷新登录状态的功能现在有2个地方实现。
第一个途径：在账号管理界面有“刷新登录状态”按钮。点击按钮后，可以批量刷新账号列表中已登录账号的状态；
第二个途径:右键已登录账号，点击“刷新登录状态”选项，执行选中账号的登录状态刷新。

刷新登录状态为独立模块。账号在线还是离线的具体判断原理在各自平台的插件中，插件中包含完整的登录状态判断逻辑和结果输出。

若点击账号管理界面的刷新登录状态按钮，则判断账号列表中已登录账号所属平台，然后调用对应平台的插件中的登录状态判断逻辑，并把判断结果返回给刷新登录状态模块，刷新登录状态模块再返回给UI，UI再更新账号列表中对应账号的登录状态。

若右键已登录账号，点击“刷新登录状态”选项，执行选中账号的登录状态刷新。刷新登录状态模块接收选中账号的平台信息，然后调用对应平台的插件中的登录状态判断逻辑，并把判断结果返回给刷新登录状态模块，刷新登录状态模块再返回给UI，UI再更新账号列表中对应账号的登录状态。

发布列表功能流程说明：

当用户点击发布列表中的“发布”按钮后，会执行以下操作：
1、查看任务列表中排在第一的任务，查看其平台及账号名称，打开浏览器跳转到其平台并加载cookie，并跳转到发布页面
2、根据账号所属平台，调用对应平台的插件，插件执行发布操作，插件执行发布操作后，将执行结果返回给发布列表模块，发布列表模块再返回给UI，UI再更新任务列表对应任务的状态。若状态为发布成功，则该任务移动到发布记录页面，若状态为发布失败，则该任务保留在发布列表页面，并更新任务状态为发布失败。

所有平台的发布逻辑都存储在对应平台的插件中，发布列表模块只负责调用插件执行发布操作，并将执行结果返回给UI，UI再更新任务列表对应任务的状态。基于以上逻辑，帮我生成发布列表功能优化文档。存放在
