> **文档版本**：v1.0  
> **适用版本**：重构后架构 v2.x（插件化架构落地后）

> **文档状态**：草案（Draft）

## 1. 产品概述

### 1.1 产品简介

**媒小宝（WeMediaBaby）** 是一款基于 Windows 的桌面端自媒体管理与自动化发布工具，面向中小团队及个人创作者，聚合多平台账号，统一管理内容素材与发布流程，降低运营成本与被风控风险。

### 1.2 解决的问题

- **多平台运营成本高**：创作者需要频繁在抖音、快手、小红书、视频号等平台之间切换登录、上传素材、填写标题标签，重复且易出错。
- **多账号管理混乱**：同一平台存在多个账号，cookie 与登录状态分散在不同浏览器/设备中，缺乏统一视图与批量管理能力。
- **自动化能力不足**：缺少稳定可靠的自动化发布与批量任务调度能力，难以规模化运营。
- **风险与安全问题**：简单脚本/第三方工具存在高封号风险，且可能泄露账号 cookie 或密码。

### 1.3 产品目标

- **提升效率**：用统一界面与自动化流程，将多平台、多账号的日常操作标准化、模板化。
- **控制风险**：通过抗检测浏览器、指纹策略和合理并发控制，在效率与安全之间取得平衡。
- **可扩展性**：通过插件化架构支持平台扩展与商业化插件（Pro）。
- **便于团队协作**：提供可视化的发布队列、发布记录和基础数据，为运营决策提供支撑。

### 1.4 目标用户

- **个人创作者 / 工作室主理人**：运营 2–10 个自媒体账号，需要稳定、高效的日常发布与维护工具。
- **MCN / 小型运营团队**：管理多账号矩阵，需要批量发布、任务队列与权限控制。
- **技术型内容运营者**：有一定自动化意识，希望在可控风险下扩展自己的工具链。

## 2. 用户画像与使用场景

### 2.1 用户画像

1. **个人创作者 A（初阶）**
   - 账号规模：抖音/快手各 1–2 个账号。
   - 诉求：简化登录、刷新状态、日常发布流程；希望“少点几步就能发完”。
   - 关注点：UI 友好、操作简单、不要太“技术感”。

2. **工作室 B（小团队）**
   - 账号规模：多个平台合计 10–30 个账号。
   - 诉求：集中管理账号状态、快速切换账号、批量任务发布，减少人工排期工作。
   - 关注点：批量能力、稳定性、账号安全（不能轻易被封或丢号）。

3. **MCN C（成长型）**
   - 账号规模：几十到上百个账号。
   - 诉求：可扩展的批量任务、基础数据可视化（未来）、团队协作。
   - 关注点：扩展能力（插件）、并发与风控策略、未来是否能对接云控/分布式。

### 2.2 核心使用场景

1. **多账号集中管理**
   - 用户在“账号管理”页查看所有平台账号，一眼看到登录状态（在线/离线）、最近登录时间。
   - 一键刷新登录状态，批量检查已登录账号的实际可用性。
   - 添加新账号时，通过内置浏览器引导用户登录，对 cookie 进行本地加密存储。

2. **内容单条发布**
   - 在“发布”页面选择目标平台与账号，选择本地视频/图片素材，填写标题、文案、标签等。
   - 支持一键发布，并将结果写入“发布记录”，可查看发布结果和失败原因。

3. **发布列表与任务队列**
   - 用户将若干待发内容（素材 + 文案 + 账号）加入“发布列表”。
   - 点击队列中的“发布”按钮，系统根据任务顺序与平台限制，驱动对应插件自动执行发布。
   - 任务发布成功后，自动移动到“发布记录”页；失败则保留在列表并标记失败原因。

4. **批量任务（中期）**
   - 用户定义批量任务（如“为 X 账号矩阵发布 N 条视频”），配置脚本参数与并发度。
   - 系统按配置进行调度与重试，支持查看每个子任务的状态和错误。

5. **素材管理**
   - 用户在“文件/素材页”集中查看已导入的本地素材文件（视频/图片），可查看基础属性（时长、尺寸、大小）和关联脚本。
   - 支持按关键词、文件类型和创建时间过滤。

## 3. 产品范围与版本规划

### 3.1 社区版（Community）范围

- 支持平台：抖音、快手。
- 支持功能：
  - 多账号本地管理（账号列表、添加、删除、编辑、账号分组）。
  - 内嵌浏览器与 Playwright 登录、刷新登录状态。
  - 单条发布与基础发布列表功能。
  - SQLite 本地数据库与加密 cookie 存储。
  - 基础工作台（仪表盘雏形）。

### 3.2 专业版（Pro）扩展范围（未来）

- 额外平台：小红书、视频号等。
- 扩展能力：
  - 高级批量任务与队列管理（并发、重试、定时）。
  - 更强的指纹与反检测策略。
  - 账号云端备份与多设备同步。
  - 数据统计与报表（播放、点赞、转发等）。

### 3.3 当前版本（v2.x）交付范围

本 PRD 重点约束当前重构后的 **基础能力闭环**：

- 核心闭环：  
  账号管理 → 登录/刷新状态 → 素材管理 → 单/多条发布 → 发布记录。
- 平台范围：抖音、快手（Community 版）。
- 插件架构：插件接口与开源插件目录确定，Pro 插件加载机制预留但不强制实现具体业务。

## 4. 功能需求

> 说明：本章节以模块划分功能需求，优先级采用：  
> - **P0**：MVP 必须实现  
> - **P1**：高优，MVP 后优先实现  
> - **P2**：后续优化/增强

### 4.1 账号与权限（Account & Auth）

#### 4.1.1 软件用户账号体系

- **目标**：为软件本身建立用户账号体系，用于后续订阅、设备绑定等扩展。
- **需求点**：
  1. 支持用户注册、登录、找回密码（邮箱或其他渠道以具体实现为准）。【P1】
  2. 支持基础角色字段（如 user/admin），用于区分权限。【P2】
  3. 记录用户的注册时间、最后登录时间、试用次数等，用于限制免费版策略。【P1】
  4. 所有用户密码使用安全哈希（目前已采用）存储，不允许明文落盘。【P0】

#### 4.1.2 平台账号管理（Platform Accounts）

- **目标**：在本地安全地管理多个平台账号信息，并提供统一视图与操作入口。
- **需求点**：
  1. 账号列表页展示字段：【P0】
     - 平台（抖音/快手/...）
     - 平台用户名（platform_username / 抖音号等）
     - 昵称（可选）
     - 登录状态（在线/离线/未知）
     - 最近登录时间
     - 所属账号组
  2. 添加账号流程：【P0】
     - 用户选择平台 → 打开浏览器引导登录 → 登录成功后，系统自动捕获并加密存储 cookie。
  3. 编辑账号信息：【P0】
     - 支持修改账号备注名、分组信息等（不直接修改底层平台用户名）。
  4. 删除账号：【P0】
     - 删除账号时给出二次确认，说明会删除本地 cookie 与相关配置（不影响平台真实账号）。
  5. 账号分组管理：【P1】
     - 支持创建账号组、修改组名、删除组，并将账号分配到组。

#### 4.1.3 刷新登录状态

- **目标**：在不频繁人工检查的前提下，可靠识别账号是否仍处于有效登录状态。
- **需求点**：
  1. 刷新入口：【P0】
     - 账号列表顶部“刷新登录状态”按钮：对已登录账号执行批量检测。
     - 账号行右键菜单“刷新登录状态”：对单个账号执行检测。
  2. 业务逻辑：【P0】
     - 按账号所属平台调用对应插件中的登录状态检测逻辑。
     - 批量刷新时，同一平台下的多个账号允许并行检测，需控制并发度以避免触发风控。
     - 单账号刷新时采用单线程串行逻辑，实时反馈结果。
  3. UI 行为：【P0】
     - 刷新时展示进行中状态，完成后更新登录状态字段与最后检测时间。

### 4.2 工作台（Dashboard / Workspace）

- **目标**：为用户提供整体运营情况的“总览入口”，并聚合关键操作快捷方式。
- **需求点**：
  1. 展示内容：【P1】
     - 账号总数、在线账号数。
     - 近期发布任务数量（成功/失败）。
     - 最近操作日志/提示。
  2. 快捷操作卡片：【P1】
     - “添加账号”“新建发布任务”“打开发布列表”等常用入口。
  3. 后续可扩展为数据中心（播放量、点赞等）。【P2】

### 4.3 浏览器控制与自动化

- **目标**：将浏览器自动化能力封装为服务，供账号登录与内容发布场景复用。
- **需求点**：
  1. 双引擎模式：【P0】
     - 内嵌引擎（QWebEngine）：用于日常浏览、基本维护。
     - 外置引擎（Playwright/Undetected）：用于登录与发布自动化。
  2. 浏览器控制对话框：【P0】
     - 展示当前任务状态（等待登录、登录中、检测中等）。
     - 提供手动操作入口（如用户在页面中手动点击登录）。
  3. 指纹与环境隔离：【P1】
     - 同一账号在多次登录/发布时保持浏览器指纹一致。
     - 不同账号使用独立 storage_state 与用户目录，互不污染。

### 4.4 发布系统（Publishing）

#### 4.4.1 单条发布

- **目标**：允许用户为单个账号快速上传并发布一条内容（视频或图文）。
- **需求点**：
  1. 发布表单：【P0】
     - 选择平台与目标账号。
     - 选择素材文件（本地路径，视频或图片）。
     - 填写标题、描述、标签等基础字段。
  2. 发布执行：【P0】
     - 调用对应平台 `PublishPlugin`，驱动浏览器执行上传、填写和发布操作。
     - 成功后写入“发布记录”，记录平台、账号、文件、时间、结果 URL。
     - 失败时记录错误信息与日志引用。

#### 4.4.2 发布列表（任务队列）

- **目标**：支持用户将多个待发布内容排队执行，并可按任务粒度查看与操作。
- **需求点**：
  1. 列表结构：【P0】
     - 每条任务包括：平台、账号、文件路径、标题等核心字段。
     - 状态枚举：待发布、发布中、发布成功、发布失败。
  2. 操作能力：【P0】
     - 单条任务的“发布”按钮。
     - 队列顶部“一键开始”按钮（按顺序执行）。
  3. 发布流程（结合现有实现）：【P0】
     - 点击“发布”时，系统取队列中第一条任务，调用对应平台插件执行发布。
     - 发布成功 → 将任务移入“发布记录”并标记成功。
     - 发布失败 → 保留在列表，同时更新状态与错误说明。

#### 4.4.3 批量任务（Batch Tasks）

- **目标**：在发布列表的基础上抽象出可复用的“批量任务配置”，支持队列管理与执行监控。
- **需求点**：
  1. 批量任务配置：【P1】
     - 任务名称、目标平台/账号、脚本配置（JSON）、视频数量等。
     - 最大并发数、重试次数、每条任务间延迟等。
  2. 任务执行与监控：【P1】
     - 后台调度执行子任务，实时记录每条执行记录。
     - 支持查看批量任务执行进度（已完成/失败数量）。

### 4.5 素材管理（Media Files）

- **目标**：统一管理本地媒体文件元信息，为发布流程提供稳固的数据支撑。
- **需求点**：
  1. 素材导入与扫描：【P1】
     - 从指定目录批量扫描媒体文件，或在发布时自动登记素材记录。
  2. 列表显示与过滤：【P1】
     - 支持按文件类型（视频/图片）、关键字、时间过滤。
     - 展示文件名、路径、大小、时长、分辨率等。
  3. 脚本关联：【P2】
     - 允许为媒体文件关联脚本配置文件，用于批量发布任务。

### 4.6 订阅与商业化（Subscription）

- **目标**：为未来的社区版/专业版区分与付费订阅功能预留产品模型与基础界面。
- **需求点**：
  1. 套餐模型：【P1】
     - 支持定义不同套餐（如 Free、Pro-Month、Pro-Year）及价格。
     - 订阅记录包含起止时间、订单号、支付方式等。
  2. 权限控制：【P1】
     - Free 版限制账号数量、并发数或可用平台。
     - Pro 版解锁更多平台（小红书、视频号）与批量发布能力。
  3. 支付与激活流程：【P2**
     - 接入微信/支付宝等支付渠道。
     - 客户端本地或通过服务端验证许可证状态。

### 4.7 插件系统与平台支持

- **目标**：通过标准化接口支持多平台接入，区分开源插件与闭源商业插件。
- **需求点**：
  1. 插件接口：【P0】
     - 定义 `LoginPlugin` 和 `PublishPlugin` 接口，约束最小实现集合（登录入口 URL、cookie 保存方式、发布字段映射等）。
  2. 插件发现与加载：【P0】
     - 系统启动时扫描 `src/plugins`（开源）与 `src/plugins_pro`（闭源）目录。
     - 按平台标识（如 "douyin"、"kuaishou"）选择对应插件实现。
  3. 配置与选择器热更新：【P1**
     - 平台配置（URL、域名、参数）及页面选择器存放在 `config/platforms/*.json` 与 `config/selectors/*.json`，支持更新不改代码。

## 5. 非功能需求

### 5.1 性能与资源

- 在典型场景下（同时管理 20–50 个账号，发布队列 20 条以内），应用应保持 UI 流畅，无明显卡顿。【P1】
- 数据库存取采用异步方式（已通过 `aiosqlite` 实现），避免阻塞界面线程。【P0】
- 启用 SQLite WAL 模式，确保在频繁写入发布记录、任务状态的情况下仍具备可接受性能。【P0】

### 5.2 安全性

- 平台账号 cookie 只在本地存储，并使用对称加密（Fernet）落盘，禁止上传到云端或日志中。【P0】
- 软件用户密码必须以哈希形式存储，不能回推原文密码。【P0】
- 访问第三方平台时遵守平台使用规范，避免违反服务条款的高危操作（如极端并发、暴力刷新等）。【P1】

### 5.3 兼容性

- 运行环境：Windows 10 及以上版本（桌面端）。【P0】
- 屏幕分辨率适配：至少保证 1366x768 分辨率下布局不出现严重错位。【P1】

### 5.4 易用性与可用性

- 所有核心操作路径（添加账号 → 刷新 → 发布列表 → 发布记录）应在 3–5 次点击内完成。【P1】
- 常见失败场景（登录失败、cookie 过期、发布失败）应给出明确的错误信息与可行的下一步提示（如“请重新登录账号”）。【P1】

### 5.5 可运维性

- 所有关键业务操作需在日志系统中留下可追溯记录（如账号登录、发布任务执行、批量任务错误）。【P0】
- 提供一键“工厂重置”脚本，清空本地数据库、cookie、日志等，以便开发调试和用户故障排查（已通过脚本实现）。【P0】

## 6. 约束与假设

- 假设用户具备基础电脑操作能力，能够理解平台账号登录与 cookie 关系。【前提】
- 假设各平台在一段时间内页面结构相对稳定，选择器配置可通过更新 JSON 的方式及时修复。【前提】
- 产品暂不提供云端数据同步功能，所有账号与发布记录默认仅本地存储。【约束】

## 7. 验收标准与度量指标

### 7.1 功能闭环验收

- **账号管理闭环**：可以完成账号添加、编辑、删除、分组、刷新登录状态操作，并在 UI 中正确显示结果。
- **发布闭环**：可以完成单条发布与通过发布列表启动任务，任务结果正确写入发布记录。
- **浏览器自动化闭环**：可以由 UI 触发登录/发布流程，浏览器服务按预期执行，并在关键节点更新状态。

### 7.2 质量与体验指标（建议）

- 功能回归覆盖率：核心流程（账号管理、发布、批量任务）均应有自动化或半自动化测试用例。【目标】
- 用户操作错误率：在常见路径上出现“迷路”或误触的比例不超过 5%。【目标】
- 崩溃率：正常使用场景下应用崩溃频率极低（< 0.5% 会话）。【目标】
