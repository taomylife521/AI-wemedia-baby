# 发布列表功能优化方案（基于最新流程图）

## 1. 业务流程概述

根据最新的发布任务执行流程图，发布列表的执行逻辑应变更为一个**支持自动循环处理的队列系统**。核心流程如下：

1. **触发与前置检查**：
   - 用户点击“开始”按钮。
   - 查看【发布列表】。
   - 如果发布列表中无任务，则终止发布，并提示“发布列表无任务”。
   - 如果发布列表中有任务，则进入队列处理循环。

2. **单任务处理循环**：
   - **读取任务**：读取任务列表中**第一个状态为“待发布”**的任务的【平台名称】和【平台昵称】。
   - **状态校验**：刷新该账号的登录状态。
     - **注：必须直接复用【账号库】界面已有的“刷新登录状态”功能逻辑，不要重复造轮子。**
     - **若账号掉线**：
       - 日志中显示“已经掉线，请重新登陆”。
       - 将该任务的发布状态更新为“已掉线，失败”。
       - 检查任务列表是否还有其他待发布任务。如果有，回到“读取任务”步骤继续处理下一个；如果没有，终止流程。
     - **若账号在线**：
       - 调用并打开对应账号的浏览器环境。
       - 开始执行自动化发布任务。
       - **发布成功**：
         - 将该任务标记为“已发布”。
         - 将任务从【发布列表】移动到【发布记录】页面。
         - 发布状态标记为“成功”。
         - 检查任务列表是否还有其他待发布任务并循环处理。
       - **发布失败**：
         - 标记该发布任务为“失败”（保留在发布列表中不移动）。
         - 检查任务列表是否还有其他待发布任务并循环处理。

## 2. 核心优化点与重构建议

### 2.1 引入队列调度机制 (Queue & Scheduler)

目前的代码更多是单次触发，缺乏连续处理多个任务的能力。需要引入一个基于协程的轻量级任务调度器。

- **机制**：点击“开始”后，启动一个后台常驻的 `while True` 或递归异步函数。
- **职责**：每次循环从数据库（或内存列表）中 fetch 出第一条 `status='pending'` 的记录。

### 2.2 严格的前置状态校验

在真正启动笨重的 Playwright 浏览器之前，必须执行状态检查。

- **复用与模块化解耦**：
  - 调用并复用【账号库】界面已有的“刷新登录状态”功能逻辑。
  - **若该功能当前不是独立模块，则将其抽离为独立模块**。
  - **不同平台的刷新原理存在差异，各自的实现逻辑必须存放在对应平台的插件文件夹中**，做到可插拔、方便管理与更新。
- **失败快速流转**：如果检测到掉线，不启动发布脚本，直接更新状态为“已掉线，失败”，避免浪费时间启动浏览器，然后迅速进入下一个任务。

### 2.3 异常隔离与连续执行

- **异常捕获**：每一个单任务的执行必须被包裹在独立的 `try...except` 块中。
- **状态流转**：无论是成功、业务失败（如被风控拦截）、还是系统异常（如定位不到元素），都必须有明确的出口回写数据库，**绝对不能因为单一任务的崩溃导致整个发布队列挂起**。完成当前任务（无论成功失败）后，通过 `finally` 块确保释放资源（如关闭无用的浏览器 Tab），然后读取下一个任务。

### 2.4 UI 响应与日志反馈

- **实时日志**：在日志面板中清晰地输出“开始处理 [平台] [账号]”、“检测到掉线，跳过”、“处理成功”等状态。
- **状态联动**：由于是自动化批量处理，UI 表格需要通过 Signal 插槽机制，在每次数据库状态改变时，实时刷新当前任务的行数据显示，而不是等所有任务跑完才刷新。
