# 3.1 抖音插件功能介绍

> **所属分类**：社区插件（开源）
> **源码位置**：`src/plugins/community/douyin/`

本篇文档介绍抖音（Douyin）平台的登录与发布自动化插件实现。发布流程采用**步骤链 + 运行器**，步骤文件按「步骤序号 + 用途」命名，便于对照与维护。

---

## 1. 文件结构

```text
src/plugins/community/douyin/
├── __init__.py
├── config.json              # 平台元数据与 limits（标题/描述字数等）
```

## 2. 登录插件 (`DouyinLoginPlugin`)

抖音登录插件继承 `LoginPluginInterface`，实现浏览器内在线检测、用户信息提取与 HTTP 静默验证。掉线原因由 `LoginResult.error_message` 提供，主程序会写入任务失败原因并在日志/UI 展示。

### 2.1 浏览器页面在线检测

由 `check_login_status` 方法执行。当浏览器当前页面位于 `douyin.com` 或 `amemv.com` 域名下时：

- 先检查 Cookie：若无关键 Session Cookie，直接返回未登录。
- 再结合 DOM：使用 `selectors.py` 中的用户昵称/头像等选择器在页面上探测；命中则视为已登录。
- 若需进一步兜底，可配合 `scripts.py` 中的 `LOGIN_DETECTION_SCRIPT` 等脚本结果综合判断。

### 2.2 用户信息提取

由 `extract_user_info` 方法执行：

- 获取当前页面的所有 Cookies，转换为标准字典格式。
- 优先从 DOM（昵称/头像选择器）或页面全局变量（如 `__INITIAL_STATE__`、`userInfo`）中提取昵称。
- 若提取失败则返回 `LoginResult(success=False, error_message="未能提取到昵称")`。

### 2.3 纯 HTTP 静默验证 (`verify_cookie_http` / `verify_account_status`)

适用于不启动浏览器的后台静默探活（如主程序发布前“刷新登录状态”）：

- **请求地址**：`https://creator.douyin.com/aweme/v1/creator/pc/user/info/`
- **有效性判定**：接口返回 `status_code == 0` 且包含 `uid` 或 `user_info` 时视为有效；失败时返回 `LoginResult(success=False, error_message=...)`，主程序据此标记任务“已掉线，失败”。

## 3. 发布插件 (`DouyinPublishPlugin`)

发布插件继承 `PublishPluginInterface`，在 `publish()` 内按 `metadata["file_type"]`（video/image）组装步骤链，交给 **StepRunner** 顺序执行。步骤可返回 `None`（继续）、`PublishResult`（结束）、`NeedsAction`（补救后重试提交）。

### 3.1 表单字段定义

`get_form_schema(content_type)` 返回发稿表单字段；`max_length` 等从配置 limits 读取（优先 `config/platforms/douyin.json`，否则插件 `config.json`，否则默认值）：

| 字段名        | 类型     | 说明                           |
| ------------- | -------- | ------------------------------ |
| `title`       | 短文本   | 作品标题（含 max_length）      |
| `description` | 多行文本 | 作品描述/文案（含 max_length） |
| `tags`        | 标签列表 | 发布时的 `#话题` 标签（选填）  |

### 3.2 发布步骤与代码文件对照表

发布流程为 **8 步主链**，视频与图文在步骤 5、6 有分支。下表严格对应实际执行的步骤与代码文件。

| 步骤      | 代码文件                 | 步骤类名                | 说明                                                                     |
| --------- | ------------------------ | ----------------------- | ------------------------------------------------------------------------ |
| 1         | `step_01_home.py`        | `NavigateHomeStep`      | 导航至创作者中心首页，做登录/风控检测                                    |
| 2         | `step_02_entry.py`       | `EnterPublishEntryStep` | 从首页点击「发布视频」或「发布图文」，进入对应发布页                     |
| 3         | `step_03_upload.py`      | `UploadMediaStep`       | 根据 file_type 上传视频或图文素材，等待就绪                              |
| 4         | `step_04_description.py` | `MetadataFillStep`      | 填写标题、描述与话题标签                                                 |
| 5（视频） | `step_05_cover_video.py` | `CoverVideoStep`        | 视频封面：首帧/本地图/AI，按任务配置分支                                 |
| 5（图文） | `step_05_cover_image.py` | `CoverImageStep`        | 图文封面：首张/本地图/AI                                                 |
| 6（视频） | `step_06_extra_info.py`  | `ExtraInfoCommonStep`   | 扩展信息：补充信息弹窗等（视频直接此步）                                 |
| 6（图文） | `step_06_music.py`       | `SelectMusicStep`       | 选择音乐（占位）；然后 `step_06_extra_info`                              |
| 7         | `step_07_settings.py`    | `PublishSettingsStep`   | 发布设置：定时发布等（占位 + 兜底）                                      |
| 8         | `step_08_submit.py`      | `SubmitStep`            | 点击发布并校验结果；遇「需要封面/补充信息」返回 NeedsAction 由运行器补救 |

**辅助文件（非步骤）：**

| 角色   | 代码文件         | 说明                                                                                           |
| ------ | ---------------- | ---------------------------------------------------------------------------------------------- |
| 基类   | `_base.py`       | `BasePublishStep`、`NeedsAction`、`StepOutcome`，所有步骤继承用                                |
| 运行器 | `step_runner.py` | `StepRunner`、`RunnerConfig`：按顺序执行上述步骤，每步可重试，NeedsAction 时执行补救后重试提交 |

### 3.3 自动发布流程说明

- **组装顺序**（在 `publish_plugin.py` 中）：  
  首页 → 进入发布页 → 上传 → 描述 → **封面**（视频用 CoverVideoStep / 图文用 CoverImageStep）→ **扩展信息**（视频仅 ExtraInfoCommonStep；图文为 SelectMusicStep + ExtraInfoCommonStep）→ 发布设置 → 提交。

- **StepRunner 行为**：
  - 仅当上一步返回成功（`None` 或 `PublishResult(success=True)`）才执行下一步；任一步失败则返回 `PublishResult(success=False, error_message=..., failed_step=...)` 并退出。
  - 每个步骤支持可配置重试（默认最多 3 次），便于应对网络/卡顿。
  - **NeedsAction**：提交后若需封面或补充信息，Runner 调用对应 handler 执行补救步骤，再回到 SubmitStep 重试，避免一次弹窗就整体报错。

- **封面步骤**：支持任务配置 `cover_type`（first_frame / custom / ai）与 `cover_path`（本地封面图路径）。在近期重构中，通过引入 `WeMedia X-Ray` 分析工具日志探针辅助，已实现对新版抖音创作者中心图文（Image）和视频（Video）在 DOM 结构深层嵌套中的精准靶向选取与上传触发，详情参见步骤 5。

- **防风控**：发布流程已接入项目**发布层防风控**（公共 anti_risk 模块），与主程序使用的**浏览器层防风控**（Undetected Playwright + 指纹 + stealth）在同一链路中同时生效。配置与对接说明见下文 3.4；完整机制见《防风控机制设计》。

### 3.4 发布层防风控对接说明

项目防风控分为**浏览器层**（环境级抗检测）与**发布层**（行为级延迟/拟人/冷却），抖音发布插件在发布层已完成对接。

**配置来源**：`config/platforms/douyin.json` 中的 `anti_risk` 块（如 `step_interval_base_seconds`、`operation_delay_min_seconds`、`enable_human_like`、`cooldown_after_frequent_seconds` 等）。发布插件在 `publish()` 入口读取并写入 `metadata["anti_risk_config"]`，供各步骤与 StepRunner 使用。

**StepRunner 行为**：

- 每步执行前：调用 `optional_browse_before_action(page, metadata, config)`，按概率做随机鼠标移动或轻微滚动，增强流程自然化。
- 每步成功后：调用 `step_interval(page, metadata, config)`，步骤间加入基准 + 随机的间隔，避免固定节奏。

**各步骤中的使用**：

- **步骤 2（进入发布页）**：入口按钮使用 `human_click`，等待使用 `random_delay`。
- **步骤 3（上传）**：上传完成轮询等待使用 `random_delay`。
- **步骤 4（作品描述）**：标题使用 `human_type_text`，描述区点击使用 `human_click`，固定等待使用 `random_delay`。
- **步骤 5～7（封面/扩展信息/发布设置）**：关键点击使用 `human_click`，固定等待使用 `random_delay`。
- **步骤 8（点击发布）**：发布按钮使用 `human_click`，提交前后等待使用 `random_delay`；检测到「操作频繁/风控」时先调用 `cooldown_before_retry(平台配置秒数)`，再返回 `NeedsAction(action="need_retry")` 由 Runner 重试提交。

上述能力由 `src/infrastructure/anti_risk/`（delays、human_like）与 `human_behavior.py` 提供；未提供 config 或模块不可用时步骤会回退为普通 `page.click` / `fill` / `wait_for_timeout`，不影响流程可运行性。

## 4. 辅助文件

### `scripts.py`

包含 `LOGIN_DETECTION_SCRIPT` 等注入浏览器执行的 JavaScript，用于登录态与用户信息检测的兜底。

### `selectors.py`

集中维护 DOM 选择器，按 LOGIN、HOME、PUBLISH、SECURITY、VERIFY 等分组，多候选列表便于页面改版时只改选择器而不改逻辑。失败诊断时会探测关键选择器命中数并截图到 `debug/screenshots/douyin/`。

## 5. 防风控机制对接情况

项目防风控包含**浏览器层**与**发布层**，二者在同一发布链路中同时生效。详见《防风控机制设计》；此处仅说明抖音插件在其中的位置与已对接能力。

### 5.1 浏览器层（环境级）

发布任务使用的浏览器由主程序通过 **UndetectedBrowserManager** 提供，具备：

- 启动参数去自动化特征、指纹（UA、屏幕、WebGL、硬件等）、stealth 脚本注入、no_viewport 等。
- 所在文件：`src/infrastructure/browser/browser_manager.py`、`profile_manager.py`、`stealth.js` 等。

抖音发布插件不直接配置浏览器层，拿到的 `page` 已运行在上述环境中。

### 5.2 发布层（行为级）— 抖音插件已对接

发布层能力由 `src/infrastructure/anti_risk/` 与 `human_behavior.py` 提供，抖音插件已完成对接：

| 能力           | 在抖音插件中的使用                                                                                       |
| -------------- | -------------------------------------------------------------------------------------------------------- |
| 步骤间间隔     | StepRunner 每步成功后调用 `step_interval()`                                                              |
| 步骤前随机浏览 | StepRunner 每步执行前调用 `optional_browse_before_action()`                                              |
| 随机延迟       | 各步骤内用 `random_delay()` 替代固定 `wait_for_timeout`                                                  |
| 操作级延迟     | `human_click` / `human_type_text` 内可选 `operation_delay()`（1～5 秒随机）                              |
| 拟人点击       | 步骤 2/4/5/6/7/8 关键按钮使用 `human_click()`（区域内随机坐标 + 贝塞尔移动）                             |
| 拟人输入       | 步骤 4 标题使用 `human_type_text()`，逐字节奏                                                            |
| 操作频繁后冷却 | 步骤 8 检测到「操作频繁/风控」时调用 `cooldown_before_retry()` 后返回 `NeedsAction(need_retry)` 重试提交 |

配置来自 `config/platforms/douyin.json` 的 `anti_risk` 块；`metadata["speed_rate"]` 继续控制整体节奏。

### 5.3 仍可增强的方向

- **账号级/任务级**：同一账号连续发布最小间隔、单日发布上限等，适合放在发布列表或 pipeline 层，非插件单任务内。
- **风控/滑块**：账号异常、封禁、滑块等仍为检测并中止，可后续增加「需人工验证」的明确提示与引导。
- **可选增强**：更多步骤中扩大 `human_click` / `random_delay` 覆盖，或按需调大 `anti_risk` 中的延迟/间隔参数。
